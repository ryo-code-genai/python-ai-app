# StreamlitとGemini APIで作るAIライティングツールのコード解説

この `app.py` は、Streamlitで画面を作り、Google Gemini APIに文章生成を依頼する個人用AIライティングアプリです。

ブログ記事作成、メール返信、要約、リライト、SNS投稿、企画出し、SEO案、校正などを、1つの画面から使えるようにしています。

## 全体像

このアプリの処理の流れは次の通りです。

1. ユーザーがサイドバーでモデルやツールを選ぶ
2. フォームにテーマや目的を入力する
3. 入力内容からプロンプトを組み立てる
4. Gemini APIに送信する
5. 生成結果を画面に表示する
6. 結果を履歴に保存し、Markdownでダウンロードできるようにする

中心になっているのは、次の3つです。

- 入力フォーム
- プロンプト生成
- Gemini API呼び出し

## ツール定義の仕組み

冒頭では `ToolSpec` というデータ構造を定義しています。

```python
@dataclass(frozen=True)
class ToolSpec:
    key: str
    name: str
    description: str
    button_label: str
    builder: Callable[[dict[str, str]], str]
```

これは各ツールの設定表です。

例えば「ブログ記事作成」というツールには、表示名、説明、ボタン名、そしてプロンプトを作る関数が紐づいています。

実際のツール一覧は `TOOLS` にまとめられています。

```python
TOOLS = {
    "blog": ToolSpec(...),
    "email": ToolSpec(...),
    "summary": ToolSpec(...),
}
```

この設計のおかげで、新しいツールを追加するときも、プロンプト生成関数を1つ作って `TOOLS` に登録すれば拡張できます。

## プロンプト生成関数

`build_blog_prompt`、`build_email_prompt`、`build_summary_prompt` などは、ユーザー入力をGeminiに渡す指示文へ変換する関数です。

例えばブログ記事作成では、次のような入力を受け取ります。

- 目的
- テーマ
- 想定読者
- 文体
- 文字量
- 含めたい要素

そしてGeminiに対して「こういう形式で出力してください」と明確に依頼します。

この部分が、このアプリの品質を決める重要な箇所です。AIアプリでは、画面よりも「どんなプロンプトを作るか」が結果に大きく影響します。

## Gemini APIの呼び出し

Geminiへの送信は `generate_text` が担当しています。

```python
response = client.models.generate_content(
    model=model,
    contents=prompt,
    config=types.GenerateContentConfig(...)
)
```

ここで指定している主な設定は次の通りです。

- `model`: 使用するGeminiモデル
- `prompt`: ユーザー入力から作った指示文
- `temperature`: 創造性
- `max_output_tokens`: 最大出力量
- `thinking_config`: 思考レベル
- `system_instruction`: アプリ全体の基本方針

現在の既定モデルは `gemini-2.5-flash` です。サイドバーでは `gemini-2.5-flash` と `gemini-2.5-flash-lite` を選べます。

## APIキーの扱い

APIキーは `get_api_key` で取得しています。

優先順位は次の通りです。

1. サイドバーに入力されたキー
2. Streamlit secrets の `GEMINI_API_KEY`
3. 環境変数 `GEMINI_API_KEY`

コード内にAPIキーを直接書かない設計になっているのは良い点です。秘密情報をGitに混ぜないためです。

## 画面構成

画面は主に2つに分かれています。

### サイドバー

サイドバーでは、次の項目を操作します。

- Gemini APIキー
- 使用モデル
- 創造性
- 最大出力量
- 思考レベル
- 使用するツール
- 生成履歴

### メイン画面

メイン画面では、次の内容が表示されます。

- 選択したツール名
- ツールの説明
- 現在の設定
- 入力フォーム
- 生成結果
- Markdownダウンロードボタン
- 実際にGeminiへ送ったプロンプト

## フォームの出し分け

フォームの出し分けは `render_tool_form` が担当しています。

例えば、メール返信、要約、リライト、校正では「元になる文章」を入力します。

一方で、ブログ記事作成、SNS投稿、企画出しでは「テーマ」を入力します。

このように、選んだツールによって必要な入力項目を変えています。

## 入力チェック

`validate_values` は最低限の入力チェックをしています。

文章ベースのツールでは、元文章が必須です。

対象のツールは次の通りです。

- メール返信
- 要約
- リライト
- 校正

テーマベースのツールでは、テーマが必須です。

また、どのツールでも「目的」は必須になっています。

## 履歴機能

生成後は `add_history` でセッション内の履歴に保存されます。

保存される内容は次の通りです。

- 使用したツール名
- 生成時刻
- 生成結果

これはブラウザを開いている間だけの簡易履歴です。データベースには保存していません。

## このコードの良いところ

このアプリは、小さなコードながら構成が分かりやすいです。

特に良い点は次の通りです。

- ツールごとのプロンプト生成が関数で分かれている
- `ToolSpec` によってツール情報を整理している
- APIキーをコードに直接書かない
- 生成結果をMarkdownでダウンロードできる
- 実際に送ったプロンプトを確認できる

特に `ToolSpec` と `TOOLS` の設計は便利です。ツールをデータとして管理しているため、新しい文章生成メニューを追加しやすくなっています。

## 改善するなら

今後アプリを大きくするなら、`app.py` を分割すると読みやすくなります。

例えば、次のような構成が考えられます。

```text
app.py
prompts.py
tools.py
gemini_client.py
tests/
```

それぞれの役割は次の通りです。

- `app.py`: Streamlit画面
- `prompts.py`: プロンプト生成関数
- `tools.py`: `ToolSpec` と `TOOLS`
- `gemini_client.py`: Gemini API呼び出し
- `tests/`: 自動テスト

また、プロンプト生成関数はテストしやすいので、まずは `tests/test_prompts.py` を作るとよいです。

## まとめ

この `app.py` は、Streamlitを使ってUIを作り、ユーザー入力をプロンプトに変換し、Gemini APIで文章を生成するアプリです。

重要なのは、単にAPIを呼んでいるだけではなく、用途ごとにプロンプトを丁寧に作り分けている点です。

そのため、ブログ、メール、要約、校正といった複数の用途を、1つのアプリで自然に扱える構成になっています。
